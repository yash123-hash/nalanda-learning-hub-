Misconceptions About Becoming a Software Developer

Now, whether we realize it or not, software development is an essential part of our everyday lives.

Software in your car, software in your phone, or software in your fridge: It's everywhere, and it's only going to become more ubiquitous as time goes on.

But with this increase in demand for software, comes an increase in demand for software developers.

A software developer is someone who designs, creates, tests and maintains the software that we use every day. And while the demand for software developers is high, there are still a lot of misconceptions about what it takes to become one.

Myth #1: You need to be a genius.

One of the biggest misconceptions is that you need to be some sort of genius to become a software developer. The stereotype is perpetuated by movies and TV shows that depict software developers as brilliant but socially awkward geniuses.

While it certainly helps to be smart, there are plenty of software developers who aren't geniuses. Most of us are just normal people who are good at problem solving and have a knack for computers.

So no, you don't need to excel in every academic subject to be successful. What's more important than your grade point average is your willingness to learn new things and your ability to think outside the box.

Myth #2: You need to major in Computer Science.

While a Computer Science degree can certainly help, there are plenty of software developers who don't have one.

According to a 2021 developer survey by StackOverflow, questioning over 80,000 developers, around a quarter of respondents have less than a bachelor's degree in any field.

Learning the skills required to be a software developer can be done in many ways. There are plenty of coding bootcamps and online courses, but if you want to get a Computer Science degree, I've got some good news.

You don't need to spend 4 years and a ton of money getting one. In fact, there are now plenty of schools that offer accelerated, 100% online Computer Science degrees that won't cost you an arm and a leg, and can be completed in as little as two years.

Myth #3: It's too late to start.

A lot of people think that software development is a field for young people. And while it's true that many software developers start their careers in their twenties, it's never too late to start.

There are many software developers who have started their careers later in life, and there is no reason you can't be one of them. Software development is a field that values experience and creativity over age.

Whatever you've done for work up until this point has built skills that transfer: communication, staying calm under pressure, and analytical thinking—all useful as a developer.

Myth #4: You need to be a math genius.

A good programmer needs strong problem-solving skills, logic and a lot of patience. However, serious math skills are rarely required.

Know core algorithms and data structures; beyond that, basic algebra usually suffices unless you work on specialized domains requiring advanced math (unlikely in your first few years).

Myth #5: You need to be a Jack/Jill of all trades.

Trying to become a master of everything is a fast path to being average at most things. It's better to be really good at a few areas.

Focus on skills relevant to your target role. If you’re starting out, pick a language aligned with your field:

  • Web: JavaScript
  • Mobile: Swift, Kotlin, or Java
  • Unsure: any modern language to build foundations

Once you have a strong base, learning new languages and tools becomes much easier.

Myth #6: You are done learning.

Tech evolves fast. New languages, frameworks, SDKs, and platforms appear; others fade. Staying current is part of the job.

Myth #7: Your day-to-day job will be boring.

Developers wear many hats: problem solver, analyst, designer, tester. It’s creative and challenging; you have room to grow and explore.

Myth #8: Your job will always be exciting and challenging.

Some days are thrilling; others are repetitive. You’ll juggle projects, fix builds, and sit through all-hands. The good days outweigh the bad.

Myth #9: All software developers work in offices.

While some work in traditional offices, remote and hybrid work are common. With the internet and cloud tools, you can contribute from almost anywhere.

Myth #10: All software developers work on big projects.

Developers build everything from tiny tools to massive platforms—freelance gigs, startups, SMBs, enterprises, and personal projects all count.

Debunking Software Development Myths

Now that we have broken the stereotypes about becoming a programmer, let's debunk some software development myths.

Myth #1: Software development means writing a lot of source code

Indeed, typing code is important. Yet there are plenty of other activities involved in the software development process.

Other Activities

  • Meetings. They may eat up a significant portion of a software developer's workday. Whether we like it or not, meetings will happen. Team habits, company culture, the chosen development methodology, and various other factors can influence the length and frequency of meetings.
  • Communication. You'll need to respond to emails, phone calls, messages on Slack, Skype and so on. Also, agile methodologies require a fair amount of communication between team members.
  • Software quality assurance. You'll implement and execute all kinds of tests, for example, unit tests, performance tests, usability tests, user interface tests, and so on.
  • Documentation. Writing and reviewing documents is also an integral part of a software developer's work. For instance, you'll need to document your code. It may sound boring at first, but code comments, if done right, are one of the most useful resources produced by developers for developers. Trying to find out what code actually does can become a daunting task if there are zero comments. As your job responsibilities grow, you'll need to write various technical documents like developer guides or design documents.

I'm going to talk in more detail about some of these activities in the upcoming module.

Myth #2: Customers always know what they want

Contrary to popular belief, this is rarely the case. In many projects, clients come with only a nebulous idea of their desired product, which can lead to significant challenges.

Example: Vague Job Posting (rendered as code)

Upwork Posting (sample) "Need app like Uber but different. Budget flexible. Time ASAP. Should work on iOS/Android/Web. Must be secure. Looking for best price."

How to Handle It

  1. Create draft plans and visual mockups before implementing. This saves time, energy, and frustration.
  2. Have frequent checkpoints and reviews with the customer so the software evolves as envisioned. If misunderstandings occur, you only discard work since the last review.
  3. Be honest about factors that affect scope, deadlines, and budget—unexpected issues, technical debt, or resource shortages.

Myth #3: Software estimates are highly precise

It's a common misconception that development timelines can be estimated with high accuracy. Estimations are crucial because they guide decisions for both the client and the development team, but they are inherently uncertain.

Treat estimates as ranges, revisit them at checkpoints, and communicate uncertainty early.

Activities Involved in Software Development

software development is more than just writing code.

It involves numerous activities that come together to create a functional product.

Let's have a closer look at the most frequent activities involved in the process of creating software products.

- Collecting and Analyzing Requirements

Requirements can come from various sources. A customer's vision, a colleague's input, or another team's feedback. They might range from crystal clear to utterly nebulous. In a recent project, we navigated through a maze of vague client requests to finally understand and deliver exactly what was needed.

- Validating Ideas and Performing Feasibility Studies

Even with clear requirements, some ideas might not be technically feasible. Constraints like technology limitations, resource availability, or expertise gaps can be roadblocks. It's crucial to assess the viability of these ideas early on. Remember, a seemingly straightforward request can turn into a Herculean task if not vetted for feasibility. For instance, we once had to shelve an innovative feature due to resource constraints — a tough but necessary decision.

- Defining the Architecture

Architecting a software system is akin to drafting a building's blueprint. It involves outlining the structure and interactions between components, creating prototypes, and doing proof of concept. Work is sometimes required, especially if the team has no prior experience with a given technology. For example, while working on a cloud based application, we had to prototype extensively to ensure our architecture could handle the anticipated load.

- Implementation and Testing

Coding is accompanied by rigorous testing, a vital step often underestimated. It's like double checking every piece of a puzzle. Testing ensures all parts fit perfectly, avoiding surprises later. Writing unit tests and conducting thorough checks are essential to catch major bugs. A lesson learned from a past project was that an overlooked minor bug in the early stages can evolve into a major headache later.

- Creating Documentation and Guides

The type and extent of documentation vary by project and methodology from API guides to user manuals. Documents are the unsung heroes, supporting both developers and end users. In one of our agile projects, well-maintained documentation significantly eased the onboarding of new team members. It was like a treasure map, guiding them through the project's complexities.

- Release, Support, and Maintenance

Post release, the focus shifts to support and maintenance. This phase is never static. It's an ongoing conversation with the users where their feedback shapes the software's evolution. For instance, shortly after releasing one of my mobile apps, I had to quickly adapt to unexpected user behavior, leading to a significant but necessary update. Each of these activities is a critical cog in the vast machinery of software development. Understanding their nuances and interplay is key to successful project execution. As we navigate through these activities, we're not just building software, we're crafting a solution that evolves with its users' needs.

Internship: Your First Step into Software Development

We've all got to start somewhere, right? And in today's tech world, that somewhere is often an internship.

It's like a playground for grown ups. Just swap out the swings and slides for keyboards and code.

So, let's unravel what an internship is all about.

Your main task as an intern is to be a learning machine. You will be dipping your toes into coding, teamwork, software tools, and even customer interactions. Expect to work on smaller tasks, maybe bugs, and if you are doing well, even a mini-feature.

Imagine walking into the office or logging in virtually for a quick morning huddle with your team. You get an assignment that is manageable but challenges you to stretch your skills. Then you dive into coding, testing, and asking for feedback. And yes, expect a lot of coffee or tea.

How to get your foot in the door

Arm yourself with core programming skills and a willingness to learn. Your next step is to find the internships themselves. Check out company websites under their 'Careers' section. You might find they are actively looking for interns. Don't overlook social media platforms, particularly LinkedIn, and if you are in college or university, keep an eye out on the career boards and listen for announcements. These places often have partnerships with tech companies, which can give you an easier path in.

Once you spot an opportunity, it's time to hustle. Tailor your resume to reflect any relevant experience or skills, and draft a cover letter that shows your enthusiasm and understanding of the company. Open source contributions or a compelling portfolio can make you stand out. Submit your application and prepare for an interview. Some companies might even ask you to solve a coding problem or two, so keep your skills sharp.

Pros and cons of internships

On the plus side, you will get hands-on real-world experience that a classroom just can't offer. Trust me, there is no better classroom than an actual workplace to understand what corporate life is all about. And let's not forget about networking. You will rub elbows with people who've been in the game for a while. These connections can be your career guides, giving you advice that's worth its weight in gold. In fact, sometimes a successful internship can even open the door to a fulltime job. I've seen it happen and it's awesome.

But hold on, it's not all sunshine and rainbows. One major downside is the money or lack of it, to be precise. Many internships won't pay you or they might offer just a small stipend. Remember, though, that you're trading that for experience and a solid entry on your CV. Also, be prepared for some less-than-glamorous tasks. You might find yourself doing work that's not exactly the stuff of dreams. That's just part of the learning curve. And keep in mind, internships usually have an expiration date. Most last just a few months, so you will need to think about your next move. It's like a short, intense course in 'What am I Getting Myself into 101.'

Future career opportunities

When it comes to future career opportunities, an internship is like a launchpad. You could go on to become a junior developer or veer into specialized roles like testing or data analysis. It's wide open from here.

the role of a junior developer.

All right, let's move on to the role of a junior developer.Trust me, it's not just a title for newbies or coding amateurs. It's an essential position that carries its own set of responsibilities.

So what does a junior developer do? In a nutshell, you will be writing code, debugging, and learning the ropes. You are the apprentice learning from the masters if you will.

You will spend time reviewing your code with other developers, learning from their feedback, and gradually improving your skills. Often you will be paired with a senior developer who will act as a mentor, providing guidance and helping you solve problems that come up.

A day in the life

Imagine a mix of coding sessions, team meetings, and yes, quite a bit of Googling or StackOverflow-ing. You're learning as you go and that's completely fine. Just remember: the more challenges you tackle, the faster you will grow.

How to become a junior developer

Well, you don't necessarily need a degree in computer science, but it helps. Certifications, bootcamps, and a portfolio to show off your skills can also go a long way. And don't forget internships. They can be your golden ticket into a junior role.

Pros and cons

On the upside, you're gaining crucial experience and honing your skills. It's like a playground where you learn the rules of the game from experienced developers, and usually it's a pretty supportive environment. Also, unlike interns, you will actually get paid.

On the flip side, the salary won't be a top tier yet. The tasks might sometimes feel repetitive and the workload can be quite heavy at times. But hey, you've got to start somewhere, right? Being a junior dev can be tough, but stick with it, and you will not only get better, but also open doors to further career opportunities.

Future career opportunities

Becoming a junior developer is your first significant step on the career ladder. From here, you can work your way up to a senior developer or software architect role. Specialize in a certain area like front-end or back-end development, or even cross over into roles like DevOps Engineer or System Analyst. Your experience as a junior dev will be the springboard for whatever you choose to do next.

Senior Developer: A Maestro of Code

Let's explore the role of a senior developer, a position that combines experience and technical skills with mentoring and strategy.

In this role, you'll be expected to go beyond coding to guide your team and contribute to architectural decisions.

At the senior level, you are not just coding, you are leading. Your duties expand beyond just writing software. You are now responsible for architectural decisions, code reviews and often mentorship of junior devs. Basically, you are the go-to person when things get tangled in code.

Your day might begin with a project management meeting discussing milestones and deadlines. Unlike junior devs, you are not just receiving tasks, you are helping to decide what those tasks should be. You will spend your day diving into complex coding problems, reviewing code, and probably doing less Googling than before. By now, you should be the one writing the guides, not just reading them.

Getting to this point has involved a lot of hard work, experience, and continuous learning. You probably have at least a few years of experience under your belt, and have demonstrated expertise in your chosen tech stack.

Before we dive into the pros and cons, let me be upfront. I want to share a balanced view here. It's not all rainbows and sunshine, nor is it all doom and gloom.

First up, let's talk money.

As a senior dev, your paycheck reflects your experience. You've earned it.

Next, you get to have a real say in how things are done. You are part of the decision-making process, influencing the direction of projects. And let's not forget about freedom. Yes, you get to turn your ideas into code without as much red tape.

Now, the flip side.

The project's success – that responsibility is on you. You are the one the manager calls when things get rough.

Another thing, you might find yourself stuck in meetings rather than actually coding. It's part of the job, but it can be a drag. And a heads-up: work-life balance can get a bit shaky. You might end up working when you'd rather be relaxing.

Quality Assurance Engineers: The Meticulous Guardians of Software Quality

Quality Assurance engineers focus on ensuring that the software product meets the required quality standards. This includes various types of testing, such as manual and automated testing, and sometimes even working alongside developers to implement testing frameworks.

The Quality Assurance team works to identify bugs, inconsistencies, or any other issues that could affect the user's experience or the software's performance, security, and other vital aspects. Their role is essential for delivering a polished, well-functioning product.

Strategic Planning & Test Scenarios

The team plans how to test the software effectively and efficiently. They create test cases designed to cover all possible user interactions, ensuring no stone is left unturned.

Documentation

The Quality Assurance team keeps meticulous records of all tests conducted, the results obtained, and any changes made during the process. This documentation is vital for understanding the software's evolution and for future reference.

Continuous Integration & Collaboration

The Quality Assurance team works closely with developers, integrating testing into every stage of the software development lifecycle. This collaboration is essential for identifying and addressing issues early, ensuring the software evolves in a stable and reliable manner.

A Day in the Life of a QA Engineer

Your typical day might start by checking new builds and analyzing them against your test cases. You may also review automated test results and have a sync up with developers regarding any new features that require testing.

You might be performing manual tests or writing automated test scripts, but your role extends beyond executing tests and writing test scripts. You are involved in the planning stages and have a say in how features should be implemented to make them testable.

To get into QA, you usually need a good grasp of software development and testing methodologies. Certifications like ISTQB Certified Tester, a standardized qualification for software testers, can also be a great asset.

Pros of a QA Role

  • Your work is the crucial last check ensuring quality and reliability.
  • Every bug you find contributes significantly to the product's success.
  • Automation turns the job into a tech playground—deeper tech means sharper skills.
  • As the “quality police,” your observations keep standards high.
  • Skilled testers are in high demand.

Cons of a QA Role

  • Repetition—rerunning the same tests can feel like Groundhog Day.
  • Deadline pressure—you're often the last checkpoint before launch.
  • Limited control—you flag issues but rely on others to fix them.

Being in QA is a blend of challenges and rewards—a balance of high impact and high responsibility. It's not just a job; it's a vital part of the software development lifecycle.

DevOps Engineer Role

What is DevOps?

The term DevOps is a matchup of two words: "development" and "operations." The role was established to bridge the gap between developers who write code and the people who manage the environment where the code runs.

Previously, these two groups operated in isolation, unaware of each other's challenges. DevOps aims to bring them together, enabling faster and more efficient results.

Core Mission

In essence, DevOps engineers work hard behind the scenes to make sure everything runs smoothly, from the development of code to its successful deployment in production.

As a DevOps engineer, you are the vital link that ensures the code developers write actually reaches the users, whether that's through an app on their phone, a program on their computer, or a website.

Your mission? Make that journey as smooth as possible. You're in charge of ensuring that code updates happen seamlessly so that new features and improvements are delivered to the users without a hitch.

Day-to-Day

Your daily routine is a blend of planning and problem solving. You establish automated workflows that help carry code from the developer's workspace to the actual product, be that a mobile app, desktop software, or something else.

And if something goes awry, you're the firefighter who steps in to diagnose the issue and put things right.

How the Code Evolves

In DevOps, you are the magician behind the curtain who ensures the code written by developers actually gets to where it's supposed to go. In this journey, the code undergoes transformations. It gets tested, optimized, and sometimes even translated into a different language that computers can understand better.

If any of these steps fail, you are responsible for getting everything back on track.

Path to Become a DevOps Engineer

Typically, you need a strong foundation in software development and operations combined with experience in system administration, cloud computing, and familiarity with automation tools.

Pros

  • Dynamic work; you interact with multiple teams.
  • You're doing something new almost every day.
  • High demand often translates to better pay.

Cons

  • Demanding role with irregular hours and high-stress situations.
  • When systems go down, you are on the hook to fix them.
  • Work–life balance can get skewed.

Career Growth

You can delve deeper into specialized areas like cloud computing or cybersecurity. Another path could lead you to team leadership or even heading up the tech side of the company.

What does a UI/UX Designer do?

A UI/UX designer is responsible for architecting the user experience. They work on everything that the user interacts with, making sure that the app or site is not only functional, but also easy to use.

This could involve designing button layouts, page flows, and even choosing colors that guide the user's eyes where they need to go. Overall, their goal is to create an intuitive and enjoyable user experience.

A Day in the Life

Imagine your workday kicking off with a sync-up with the product and development teams. Your shared workspace is bustling with activity as you review project scopes, refine wireframes, and conduct user testing.

You act as the bridge that connects the technical and visual aspects of the projects.

How do you get into this role?

Most folks have a background in graphic design or maybe even frontend development. The key is to have a portfolio that demonstrates your ability to combine form and function.

The Brighter Side

In design, the journey blends vibrant creativity with constant adaptation—bringing ideas to life while navigating changing user preferences and technological shifts.

Career Growth & Next Steps

A senior UI/UX role is a logical next step with more leadership responsibilities. You could also specialize further—becoming an expert in areas like mobile design. If management interests you, consider becoming a Design Director.

Software Architect — The Next Step

After progressing from junior to senior developer roles, you may be wondering about your next step. The role of a software architect is a great opportunity to exercise your visionary skills, and it often overlaps with the responsibilities of a senior developer.

Sound interesting? Let's explore it together.

What a Software Architect Does

The software architect is like the architect of a building. You're responsible for creating the blueprint, the technical design that the dev team will use to construct the software. Your role is to think big, designing the structure, identifying the components and defining how they all interact.

You often decide which frameworks, technologies, and approaches the team will use. You are the go-to person for ensuring technical soundness and integrity.

A Typical Day

You might spend your day discussing technical requirements, researching new technologies, sketching out architectural diagrams, reviewing code and yes, still doing a bit of coding yourself. You're the bridge between the business needs and the technical aspects of a project. Expect a lot of meetings with product owners, stakeholders, and your dev team. Trust me, it's a juggling act.

How to Get There

Typically, you'd move into this role after gaining a few years of experience as a senior developer. But on top of that, you will need a deep understanding of software design principles, a knack for problem solving, and great communication skills.

Pros and Cons

Pros

  • You're the master planner, the visionary, and it's rewarding to see a project come to life based on your designs.
  • Architecting a project from the ground up can be immensely satisfying.
  • You are the technical face of the team and your decisions hold weight.

Cons

  • Your decisions can make or break a project. Mistakes at the architectural level can be costly to fix later.
  • Since your role often blends with that of a senior developer, expect some overlap in responsibilities.
  • Depending on your decisions, you can either be the hero or the villain.

Personal Reflection

As someone who has worked as a software architect, I can totally relate to the challenges that come with decision making. It's like having a crystal ball and having to pick between equally tempting technical options, each with its own set of pros and cons.

At times, you may feel like a superstar, but at other times you may become the subject of complaints. Every small design choice you make can lead to significant changes in the code. On the other hand, if you decide to put off a decision, it could create problems for you later on.

So yes, the role gives you a lot of authority, but with great power comes great responsibility.

Career Growth

As a software architect, you are pretty high up the food chain, but there's always room to grow. Some people transition into executive roles like chief technology officer or other high ranking managerial positions. Others become industry consultants or experts in particular architectures or technologies.

role of system analyst

Now let's talk about the system analyst. The system analyst is the person who bridges the gap between what the business needs and how technology can solve those needs.

As a system analyst, your primary responsibility is to act as a mediator between the business and development teams. Your job is to translate business requirements into a language that developers can understand.

You may also conduct feasibility studies, estimate costs, and even manage projects at times. Your morning likely starts with meetings with business stakeholders. You are the link between business needs and the tech solutions that meet them.

These meetings are about listening, asking questions, and gathering requirements. After that, you head back to your desk to create technical specifications or requirements documents.

You are not just writing down what you heard. You're analyzing — maybe even running some data models — to turn business needs into actionable steps for the development team.

Then you likely spend time reviewing existing systems, coordinating with project managers, and taking part in quality assurance processes to make sure the end product is in sync with the original vision.

It's a complex role that blends soft skills with technical expertise. Most system analysts come from a business or tech background. You could start off in a junior role after obtaining a relevant degree or transition from a related role, like quality assurance.

advantages and disadvantages to this position.

Pros.

  • You're not confined to a desk.
  • You interact with different departments.
  • Your work directly impacts the success of projects.

Cons.

  • Deadlines are real and sometimes tight.
  • Your recommendations can make or break a project.

As a system analyst, your next step could be moving into managerial or strategic roles where you'd oversee larger projects or even entire departments. You could also transition into consultancy, where your broad understanding of systems and business processes is highly valuable.

This is why they need to interact with a diverse set of people, and why their schedules often include significant travel.

It's not just about being busy. It's about being deeply involved in every aspect that affects their projects and teams.

Detached leadership.

Contrary to popular belief, effective managers are highly invested in the success of a project and are far from being disengaged.

Sure, they might have their own offices, though this is becoming less common, but that doesn't mean they are out of touch.

Any role that deals with sensitive information, intricate planning, or requires periods of intense focus benefits from a bit of solitude.

Just like a writer needs a quiet room to come up with good stories, or a coder needs uninterrupted time to solve complex problems, managers use these spaces to strategize and connect the dots for the bigger picture.

It's not about being isolated, it's about creating a space where they can be most effective for their team and the projects at hand.

Authoritarian figures.

The idea that managers exist solely to impose strict rules or create obstacles is mostly a myth.

I'm not saying that all managers are perfect, or that every decision they make is spot on, but their main job isn't just about being strict or creating problems, but more about helping their teams thrive.

It's a bit like a coach who is there to help the team play better, not just blow the whistle when something goes wrong.

Bottom line

The management role is multifaceted and indispensable for the overall success of a project.

Software Development Methodology

You wouldn’t start cooking a meal without a recipe. Similarly, the development of a software product also requires some sort of step-by-step plan.

The process which describes the steps needed to build the software from start to finish is what we call software development methodology.

Like cooking a meal without a recipe, ad-hoc software development leads to problems.

Without a plan the software quality will suffer, deadlines won’t be met and costs will get out of control.

Thus, every software development company or startup uses a development process.

Over time, many different methodologies have been invented.

Since projects have different needs, choosing the right approach is not always an easy decision to be made.

In the upcoming lectures, we’re going to talk about the plan-driven Waterfall model and the change-friendly Agile approach.

Here’s a formal definition: A software development methodology describes how to organize the activities involved in the software development process.

To summarize: software development methodologies are collections of best practices. By applying these practices, your project will go smoother and you’ll increase the chance of completing it.

The Waterfall model.

The Waterfall is a linear model. It defines development steps or phases.

You start executing one step, complete it and then start the next one.

This approach gives us a steady, downward order - hence the name "Waterfall." The development process flows in cascades. Each development phase requires the previous one to be completed.

Let’s talk a bit about these phases.

First, we collect and analyze the requirements. The expected functionality of the future application must be clarified with the stakeholders. All the details must be documented thoroughly. This very first phase is probably the most important one. When done right, the Waterfall model will produce the expected outcome.

After collecting and analyzing the requirements, we can proceed to the next phase. Here’s where we define the overall design of our software. Defining the architecture is like creating the blueprint for a building. Thus, the design should be as clear and detailed as possible. The bottom line is this: the team should be able to implement the product based on this plan.

We should address questions like: "Which packages or components will form our system?" "What are the fundamental types of each component?" "How do these types interact with each other to achieve the required functionality?" "Is our software secure?" "How about performance?" "How does our software respond to errors?" "How do we handle edge cases?" "Should we extend our system in the future?" "Which third-party components do we use?" And the list can grow or shrink depending on the requirements we defined in the previous phase.

The implementation comes next. The software development phase is usually divided into smaller units. Each unit is then implemented and tested by developers. Once the development phase is completed, the product undergoes the verification phase.

During this step, the software is evaluated based on predefined criteria. We must check whether the product provides the functionality we agreed on. Tests are executed to ensure that the software works as expected. We test for functional, performance, security and usability issues. Detected problems are recorded and fixed. The process goes on until all severe bugs are fixed.

The verification phase may also bring to surface deeper bugs and critical issues that may affect the planned release. As soon as the testing phase completes and the given version is delivered, the software enters the maintenance phase.

By definition, the maintenance phase is about fixing smaller bugs. But more often than not, it may also include functional enhancements. The client may come up with new requirements that involve substantial changes. You may feel tempted to squeeze in “just one more patch” in the maintenance phase. Now, this is usually a bad idea. In this case, we need to set up a new waterfall project and repeat all the steps.

The Waterfall model is used for life-control, medical and military systems. This model requires us to clarify all the requirements and create a detailed plan upfront. We can use the Waterfall if all requirements are precisely defined and won’t change over time.

The Waterfall has received some well-deserved criticism for its inability to respond to changes. Due to its linear structure, new requirements can’t be considered at later phases of the development process. If the client changes their mind frequently, or our design misses essential aspects, we’re going to hit problems during development or testing. In such cases, we should follow a different approach.

pros and cons of the Waterfall model.

we're going to talk about the pros and cons of the Waterfall model. Let’s first see the benefits of using the Waterfall. This model requires clarity on almost every detail before writing a single line of code. This approach has a huge benefit. The detailed design document leaves no room for misunderstandings.

In order to come up with a detailed design, the architect has to perform a thorough analysis on all the features and requirements for the given the software product. While this can be challenging and time-consuming, it has the benefits of spotting most of the issues that otherwise may appear later.

By identifying the problems early on, we can reduce redesign and bug-fix costs at later stages. Note however, that no methodology can guarantee bug-free software.

The other benefit of having a detailed, up-to-date design document is that information is not lost. The team setup usually changes over time. Imagine what happened if the senior architect, who was in charge to oversee the entire software project, suddenly leaves the company.

If all the knowledge were only in his head, those who take over the project would have to reverse-engineer the project based on the source code. For large, complex projects this may take up to several weeks!

The Waterfall Model is not well-suited for abrupt changes. It is based on the assumption that both the requirements and the design can be defined at early stages. This can lead to carving in stone some assumptions which may become obsolete later on.

The customer may change their mind and ask for new features that do not fit well in the original set of features. A customer who precisely knows what they want is every software professional’s dream to come true. Unfortunately, this rarely happens in real life.

Deciphering what the customers really want can be more challenging than you’d imagine. Therefore, coming up with a precise and clear set of requirements is time consuming and even impossible in some cases. If the requirements cannot be clarified within a certain amount of time, an Agile methodology may be a better choice.

Another caveat of the Waterfall model is the big initial investment that is required to create a detailed design. Up to 40% of the overall development time may be spent on architecture definition early in the software development cycle.

Besides, the development can't go in parallel with the design definition work.

Now that we know the pros and the cons of the Waterfall model, let’s see what kind of projects is it best suited for. If you know quite exactly what has to be built, then you can safely rely on the Waterfall model. All the pros we brought up will make your life easier.

You can’t go wrong with the Waterfall if you have a product that is firm and stable. Typically mature or legacy projects which have a stable feature set fall into this category. Go for the Waterfall model if the likelihood of unexpected feature requests is low.

If the environment your software is using or relying on - like the platform, the SDK, the hardware, follows a stable, predictable path, and every information is available, then you can’t go wrong with the Waterfall.

For the Waterfall to work, preconditions must be met. This includes the capability to clarify vague requirements, ability to write detailed design documents, in-depth understanding and knowledge of UML and design patterns, quality-awareness and willingness to write clean code based on a design document.

Agile methodology .

Agile is a relatively new approach to software project management. It all began with the agile manifesto in 2001. This manifesto was an attempt to end the proliferation of methodologies that had developed.

The agile manifesto defines four values.

  • Individuals and interactions over the processes and tools. This doesn't mean that we won't use processes and tools in agile projects. We still need tools and processes, but they shouldn’t prevent us from implementing the required features or changes. Instead of enforcing people to follow a rigid process, we basically implement a process that’s adaptive and responds to changes.
  • Working software over comprehensive documentation. This doesn't mean that agile projects don't use documentation at all. We should create documentation where it provides value. There’s no need to create extensive documentation just for the sake of it.
  • Customer collaboration over contract negotiation. Don't get this wrong either. Agile projects also require contracts to manage customer expectations about costs and schedules. But unlike for plan driven projects, there is a spirit of partnership between the development team and the customer. Due to the somewhat uncertain nature of agile projects, both parties acknowledge that some requirements and details may need to be redefined or clarified further as the project progresses. It goes without saying that this kind of partnership requires collaboration and trust.
  • Responding to change over following a plan. Agile is different from plan-driven approaches, and provides more flexibility compared to the Waterfall model. The major difference is that agile welcomes changes even at the later phases of the development cycle. Some planning is also required for agile projects. But, we don't try to come up with a detailed plan for the entire project prior to starting any development activities. As a consequence we're not blocked until all the requirements are clarified and each and every question gets answered.

Now let's talk about how an Agile approach solves the problem we saw with the Waterfall. The main idea behind Agile is that we can deliver functional software iteratively instead of delivering the entire project all at once.

The work is broken up into shorter chunks called sprints. The Sprint is usually two to four weeks long. At the end of each sprint the team should provide something that's an improvement over the previous sprint’s outcome. This interactive approach provides an opportunity to frequently review the product that’s being developed. Stakeholders have a chance to evaluate the software and provide their feedback early on rather than waiting for the final product to be delivered. These frequent checkpoints are super useful as they ensure that the project evolves in the right direction.

Unlike the waterfall, agile methodologies do not separate testing from development. Testing is tightly integrated with development and the entire team owns the responsibility for the quality of the product. Also involving the business users in the development process stands at the core of agile approaches. There’s a strong relationship between the project team and the stakeholders and business users.

This model works best in situations where the requirements can't be defined upfront. Agile is a good fit for software projects that are depending on many uncertain factors and changes are to be expected. One of the big benefits of this collaborative model is that it usually leads to higher customer satisfaction. Also, team members will likely be more motivated by engaging customers directly.

Note that Agile is not a methodology but rather a way of thinking defined by the Agile Manifesto values and principles. Scrum and Kanban are examples of discrete methodologies that implement the agile approach.

pros and cons of the Agile approach

Let’s talk about the pros and cons of the Agile approach The Agile Model takes an iterative approach to development with regular feedback intervals. This makes it possible to consistently produce quick results.

These quick iterations give the team the ability to adapt to new requirements or market changes quickly. Obviously abrupt changes need more effort than subtle ones.

Customers are usually happy for being involved in the software development process. Although this may have additional costs, the benefits are high. The customer can track and influence the evolution of the software product. This guarantees that less amount of work must be thrown away or rolled back if some requirements are misunderstood.

Drawbacks & Pitfalls

Agility doesn’t come without some drawbacks and pitfalls, though.

Estimations are often a guess-work. For large, complex software products, it is challenging to estimate the required time and effort, especially at the beginning of the software development lifecycle. Often, numbers are given based on individual gut-feeling estimates.

Customer involvement and close collaboration between team members is time-consuming. Besides, team members should have the willingness and skills to properly communicate with the customer and with each other. Members of agile teams should be experienced enough in order to quickly adapt to changes and to be capable of taking the required decisions. So, both professional and soft-skills are needed.

Although embracing changes stands at the core of the Agile model, frequent and abrupt changes can affect the team’s morale and productivity. The project can easily get off-track if the customer can’t clarify what they want. Too little documentation can lead to maintenance, communication and knowledge sharing issues - especially if key members leave the team.

When Agile Fits Best

With all that said, Agile Software Development is well-suited for projects where:

  • Requirements are incomplete or can't be clearly defined at early stages of development.
  • Scope can change frequently — for example, due to technology/hardware changes, or because the client is undecided.
  • Clients must be integrated in the software development process — e.g., the customer triggers some features as they progress with their own product.
  • Technology is unknown or not well-documented, and requires a lot of experimenting and proof-of-concept work.
  • Customers rarely have the required technical background to precisely define what they want from a software product.

Therefore, for many project situations, the advantages of agile development outweigh the disadvantages.

Waterfall vs Agile

Waterfall is usually perceived as rigid and bureaucratic compared to Agile methodologies.

However, both have their place.

There are cases where a plan driven methodology won’t work.

If the level of uncertainty is high and not all questions can be answered right away, then you should probably choose an agile methodology.

For other projects, a waterfall-like approach will be a better fit. And actually waterfall is still widely used.

Examples

Let me give you some examples.

When developing a weapons control system, the requirements should be clarified in advance and need to be stable. Changing the requirements midway would increase the costs considerably - and these kinds of projects are expensive anyway. A Waterfall approach makes perfect sense in this case.

And here’s another example from Upwork:

“Looking to create the next big social media platform for IOS and Android”

Now, coming up with a detailed plan based on assumptions wouldn’t make sense. The description is vague and the customer won’t be able to describe precisely what they need. This high level of uncertainty calls for an agile approach. Creating “the next big social network” will require multiple iterations.

Summary

So, to sum it up:

  • Use the Waterfall when you know what the final product should be, and clients can’t change the scope of the project.
  • Use Agile methodologies when there’s a lot of uncertainty involved, the requirements are vague or rapidly changing and clients can’t precisely describe what the end-product should do or look like.

Scrum Overview

The term Scrum was first introduced by Jeff Sutherland and Ken Schwaber in 1995.

Scrum takes the values and the principles defined in the Agile Manifesto and distills them into a simple to follow framework.

Scrum provides general guidance for managing complex product development, but it does not define detailed actions.

Instead, it focuses on a team-driven, empirical process.

How does this look in practice?

Let's say you are working on a new product.

The first step is to come up with an idea for the product. This can be done by talking to potential customers, doing market research, or simply coming up with a good idea on your own.

The next step is to define what the product should do. This is called the product backlog, which is a list of all the features and functionality that need to be included in the product.

The product backlog is maintained and prioritized by the product owner who's responsible for ensuring that the product meets the needs of the customer.

Roles in Scrum

In addition to the product owner, there are two other key roles in Scrum: Development Team and the Scrum Master.

The development team is responsible for building the product. They have all the skills needed to build a working product, including analysis and design.

The Scrum Master helps the team apply Scrum and follow its processes. The Scrum master makes sure that the team has all it needs to be as productive as possible.

Working with Sprints

Now let's talk about how work is organized in Scrum. Scrum uses fine grained timeboxes called sprints. Sprint is a fixed amount of time, typically 2 to 4 weeks long during which specific work is done.

Before the start of each sprint, there is a planning meeting where the product owner and the development team members decide which product backlog items to select for the given sprint.

The development team analyzes the high level items and converts them into a detailed list of tasks. The selected product backlog items and the detailed task list get merged into the Sprint backlog document.

The development team is responsible for implementing these tasks and delivering the corresponding functionality. Each team member picks a task from the Sprint backlog and starts working on it.

Development team members then meet each day to assess their progress. The Daily Scrum, also called Daily Stand Up, is a 15-minute meeting where the development team discusses what they've been working on and the obstacles that have surfaced and what is going to be worked on next.

The Scrum master and the product owner work with the team to remove any roadblocks or impediments if the sprint is slipping behind. Tasks are re-estimated and reprioritized.

Every sprint should result in a potentially shippable product increment, meaning that the product contains new features to show to customers at the end of each sprint.

There is a sprint review meeting for the development team to showcase what they have built. This usually happens in the form of a product demo. The customers and other stakeholders get a chance to check out the new features and provide their feedback.

The Sprint review meeting is followed by a retrospective meeting that goes over what was done during the sprint and what could be improved. This meeting is not about finger pointing, but instead about identifying issues and coming up with constructive solutions.

Then the process starts over again until the project completes.

Sprint planning

Sprint planning is where the development team, including the product owner, discusses the prioritized backlog of work known as product backlog items and decides which ones to complete in the upcoming sprint.

Each product backlog item is written as a user story, which helps to clearly communicate the desired functionality from the customer's perspective.

For example, a user story may be: "As a customer, I want to be able to track my order so that I know when it will arrive."

While anyone on the Scrum team can write user stories, the product owner typically gathers input from stakeholders, analyzes customers' needs and prioritizes user stories for maximum impact.

Effective user stories result in a clear understanding of project goals and help Scrum teams stay focused on delivering value to end users.

User story template

A user story would follow this template: "As a <type of user>, I want to <desired action> so that <benefit>."

Now for the 'type of user' part, be specific. Instead of just saying "a user," try to identify a particular type such as a student or a marketing manager, because it really helps the team understand who they are building for and why.

Bad vs better example

To illustrate my point, here's a bad user story:

"As a user, I want to see ads so that the company can make money."

That doesn't sound very natural or user focused.

Instead, try something like:

"As a marketing manager, I want to see targeted ads on the website so that we can increase conversions and drive revenue."

Much better, right? With a bit of refinement, the user story became clearer and more impactful, and it also sounds more natural.

Tips

  • Keep them concise and focused on the end user's needs.
  • Be specific about the user type.
  • Describe the benefit clearly.

Back to Sprint planning

Once the Scrum team commits to completing a set of user stories during the sprint, they break them down into tasks and estimate each one's level of effort, often measured in story points.

Note that story points are not a direct representation of time. Instead, they are used to help estimate the amount of effort required to complete the user story.

A story point value of 40 may not mean 40 hours worth of work — it depends on the agreement and understanding within the Scrum team.

For instance, it may represent a task that the Scrum team agrees is about as difficult as any other task they've completed in past sprints that had a story point value of 40.

Converting story points to time

How do you convert story points to time? That's up to the Scrum team to decide.

Our team used an estimation strategy called Planning Poker, where Scrum team members held up cards with numbers representing their estimate in hours for a story's level of effort. 

Planning Poker example card values: 1, 2, 3, 5, 8, 13, 20, 40

Daily Standup (Daily Scrum)

The Daily Standup, also known as Daily Scrum, is a short, usually 15-minute long meeting where the development team members meet to discuss the progress of the current sprint.

During the Daily Standup, each team member answers three questions:

  1. What did you do yesterday?
  2. What will you do today?
  3. Are there any blockers to report?

These are the only questions that should be answered. If there are other questions that need to be discussed, then they should be saved for another time. The Daily Standup is not the time for long, drawn-out discussions. The goal is to keep the meeting focused on progress and blockers.

Example Update

  • Yesterday I worked on the login page.
  • I almost finished the frontend code, but I need to do some testing before it's done.
  • Today I'm going to finish the login page and start working on the signup page.
  • The only impediment is that I'm waiting on the design for the signup page.
  • I'll follow up with the designer today.

You can clearly identify what was done yesterday, what needs to be done today and any blockers. The Daily Standup is not a time for solving problems. If there's an impediment, then it should be noted and someone should be assigned to follow up on it. But the actual problem solving should happen after the standup.

The Daily Standup is a great way to keep the team focused and on track. It's also a good way to identify any problems early on so that they can be addressed quickly.

Psychological Tricks to Keep It Short & Effective

Have everyone stand up

It's helpful to have everyone stand up during the meeting. After all, it's called a standup for a reason. Standing up has been shown to improve focus and keep meetings shorter.

The Scrum Ball

Use a “scrum ball” (any safe object) passed around during the meeting. Whoever holds the ball is the only one who can talk. After answering the three questions, they pass the ball randomly to someone else. The randomness increases attention because people don’t know when they’ll be called on.

Keep it short

To avoid getting sidetracked, keep answers short and to the point. For fun accountability, we had a rule: if anyone went way over their time limit, they had to buy the whole team ice cream. Silly—but effective.

Same place, same time

It’s crucial to have the standup at the same time and place every day. If the meeting keeps moving or getting rescheduled, people get frustrated, some will be late, and the meeting becomes less effective.

Sprint Review

Keep Slides Minimal, Focus on the Demo Slides are often used as a crutch and they tend to make sprint reviews longer and more boring.

If you absolutely must use slides, then keep them to a minimum. Focusing on the actual work instead of boring everyone to death with a slide deck is a much better use of everyone's time.

This reminds me of the following quote by Steve Jobs:
“People who know what they are talking about don't need PowerPoint.”

So try to briefly summarize the achievements of the sprint and move on to the demo.

Make the Demo the Star

The demo is where things get interesting. Technical demos, when presented well, can be really exciting and engaging. They can also be a great way to get feedback from the product owner and stakeholders, but all too often they are dry and boring.

participated in sprint review meetings, When done right, they can be a great way to get everyone on the same page and collect feedback. But when done wrong, they can be a complete disaster.

Sprint Retrospective

The Sprint retrospective is a meeting that's held at the end of every sprint. It's an opportunity for the team to reflect on the past sprint. The goal is to identify what went well, what could be improved and what actions should be taken to make improvements in the future.

The Scrum master facilitates the meeting and all members of the development team should participate. Neither product owners nor stakeholders should attend the retrospective, as it's meant to be a safe space for the team to openly share their thoughts and ideas.

I'm not sure whether the retrospective or the daily standup felt more awkward when I first started out with Scrum, but I can tell you that over time the retrospective has become one of my favorite meetings.

I wouldn't even call it a meeting anymore, but more of a chance for the team to get together and have an open, honest conversation about what's working and what's not. It's a chance to come up with creative solutions to problems, and it's a chance to build trust and collaboration within the team.

How to Run a Sprint Retrospective

Let's say you just completed a sprint. The Sprint review went relatively well, but there were a few hiccups here and there. The team couldn't quite meet the sprint goal and there were a couple of bugs that interfered with the demo.

In addition, one of the stakeholders asked a tough question that the team was not prepared to answer at the beginning of the retrospective. Everyone has a chance to share their thoughts on the past sprint. This can be done in a variety of ways, such as through a round-robin discussion or brainstorming.

Start with What Went Well

First, we went around the room and each person shared one thing that they thought went well in the past sprint. For instance, I thought the team did a great job of communicating with each other. I was really happy with how we handled the unexpected bug before the demo.

It's crucial to start the retrospective on a positive note. This sets the tone for the rest of the meeting and allows everyone to share their thoughts openly. The speaker should be brief, 1 to 2 minutes at most and should avoid getting into too much detail. Interruptions are not allowed at this stage.

The Scrum master or another designated team member should take notes during this part of the retrospective so that all ideas are captured. We usually use the whiteboard for this, or sometimes a text editor that was shared using a projector.

Bottom line: Make sure everyone can see the ideas that are being generated.

Discuss Improvements

After everyone has had a chance to share their thoughts on what went well, it's time to move on to the next topic: What could be improved? Just like before, each person takes a turn sharing their thoughts.

  • I think we could have done a better job of estimating tasks that involve research.
  • We need to improve our network layer to prevent sporadic bugs like the ones we saw during the demo.

These ideas should be captured in the same way as the positive ones. Again, the speaker should be brief and to the point.

Brainstorm Actions

After that, we brainstormed potential actions that we could take to improve the situation:

  • Before each review, test-run the demo while Zoom is open to catch glitches early.
  • Set aside time for research when estimating tasks.
  • Improve network layer’s unit test coverage to catch bugs before production.

Finally, the team decides on the top few actions they want to focus on for the next sprint. This helps the team focus on a few key areas rather than trying to tackle everything at once. These items are then added to the next sprint’s backlog.

What happens with the rest of the ideas that were generated? They are added to a parking lot, which is essentially a list of potential improvements that the team can come back to later.

Conclusion

So that's how we've been running our retrospectives. This simple process has helped us improve our sprints and learn from our mistakes. I hope it does the same for you.

Kanban Overview

Kanban will probably sound familiar to you if you've ever used a task management tool like Trello, Asana or any other board-based system. That's because Kanban is all about visualizing your work so that you can see what needs to be done and when.

The word Kanban means signboard or billboard in Japanese.

Origins

This visual management system was created in 1940 by Taiichi Ohno, an industrial engineer at Toyota Motor Corporation, to increase the efficiency of the company's manufacturing processes.

The basic idea behind Kanban is to make the workflow visible so that bottlenecks and inefficiencies can be identified and addressed.

Kanban Cards & Pull System

If inventory levels for a particular part fall below a specified threshold, a Kanban card is used to signal the need to produce more of that part. The Kanban card is basically a message from the consumer to the supplier that more parts are needed.

Unlike the push-based system of traditional assembly lines—meaning that parts are produced even if there is no demand for them—Kanban is a pull-based system, which means that parts are produced based on actual demand.

Why It Works

This approach eliminates overproduction and optimizes inventory levels, which leads to cost savings and increased efficiency.

From Factory to Software

In the early 2000s, a software engineer named David Anderson began adapting the Kanban manufacturing process for software development. His efforts led to the creation of the Kanban method, which is now used by software development teams all over the world.

Kanban is a flexible system that can be adapted to fit the needs of any organization.

The Three Steps

  1. Visualizing the work: Create a visual representation of your work, such as a Kanban board.
  2. Limiting work in progress (WIP): Set limits on the number of tasks that can be in progress at any given time to avoid bottlenecks and ensure timely completion.
  3. Improving continuously: Constantly evaluate and improve the way you work to identify bottlenecks and inefficiencies and find ways to eliminate them.

Tip: Start with a simple board (To Do → In Progress → Done) and gradually add WIP limits as your team gets comfortable.

Kanban board

The idea behind Kanban is pretty simple.

You have a board with columns representing the different stages of your workflow and cards depicting the tasks that need to be completed.

In a typical setup you might have columns for "Todo," "In progress" and "Done."

Each developer is assigned a task which they move from one stage to the next as they make progress.

Example Workflow

Let's say you start working on a task. Initially it would be in the Todo column. Once you start working on it, move it to the "In progress" column. When you're finished, move it to the "Done" column. It's that simple.

Imagine how valuable this simplicity can be when you have a team of developers working on a project with dozens or even hundreds of tasks. Kanban makes it easy to see what needs to be done, track the progress of each task and help identify and address bottlenecks in the workflow.

It's also worth noting that Kanban boards can be as straightforward or as complex as you need them to be. For instance, you might add additional columns for different stages of work, such as "In review," "QA" or "Deploy." The point is that the Kanban system is flexible and you can customize it to fit the needs of your team.

Scaling Kanban

Say you're working on a project with a large team. You can create Kanban boards for each developer or workflow stage. You might have one board for the frontend and another for the backend development tasks. Or you might create one board for the design tasks and another for the coding tasks. The possibilities are endless.

The important thing is that Kanban can be adapted to fit the needs of almost any organization or project, no matter how big or small.

Why Kanban Works

I love that Kanban is a simple system that can be used to great effect. It's easy to learn and implement but also extremely powerful.

I've been a huge advocate of Kanban. It is used on countless projects and it never fails to impress developers If you're looking for a way to increase your team's productivity, I highly recommend giving Kanban a try.

Preventing Infinite Loops with Max Login Attempts

In the previous lesson, we created a program that asks for user credentials. We relied on the while statement to keep asking for the username and the password until the login succeeds.

Now, what happens if the user has forgotten the password? The while statement would loop forever—this is known as an infinite loop, and it is one of the nastiest bugs in programming. As a rule of thumb, we should always provide an exit condition to avoid infinite loops.

In real-world authentication, users don’t get infinite chances. Too many failed attempts can lock you out of your account. We will implement this behavior by tracking the number of unsuccessful login attempts.

First, we create a variable called attempt and initialize it to 1. Next, we create max_attempts to store the maximum number of allowed failed logins (for example, 5). We also create a Boolean variable is_authenticated initialized to False. This will turn True only if the login succeeds.

Inside the while loop, whenever credentials are wrong, we increment attempt by 1 (using attempt += 1). Then we check if attempt is greater than max_attempts. If yes, we break out of the loop and lock the account. The break statement instantly terminates the loop.

If login succeeds, we update is_authenticated to True and display Login successful. After the loop finishes, we check the value of is_authenticated. If it’s still False, we inform the user: Your account has been temporarily locked. 

def login(username: str, password: str) -> bool:
    return username == "admin" and password == "1234"

user = input("Username: ")
pwd = input("Password: ")

attempt = 1
max_attempts = 5
is_authenticated = False

while login(user, pwd) == False:
    print("Login failed, re-enter your credentials.\n")
    attempt += 1
    if attempt > max_attempts:
        break
    user = input("Username: ")
    pwd = input("Password: ")
else:
    is_authenticated = True
    print("Login successful.")

if not is_authenticated:
    print("Your account has been temporarily locked.")

Let’s test it: entering wrong credentials five times results in the message Your account has been temporarily locked. But entering the correct credentials before reaching the limit will display Login successful.

By default, if we don’t specify start and step, the values are 0 and 1 respectively. It’s important to highlight the difference between a while loop and a for loop. The while loop executes as long as its condition is true, whereas the for loop runs a block of code a finite number of times.
